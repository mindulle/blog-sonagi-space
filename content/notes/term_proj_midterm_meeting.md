# 📅 텀 프로젝트 중간 점검 및 프론트엔드 제안 회의

## 1. 프로젝트 진행 현황 브리핑 (공유용)
현재 프로젝트는 **API-First Design** 원칙에 따라 설계와 스캐폴딩이 매우 잘 되어 있는 상태입니다.

*   **설계 완료:** Event Storming을 통한 도메인 모델링 및 기능 명세서 작성 완료.
*   **인터페이스 확정:** `specs/api/openapi.yaml`을 통해 프론트-백엔드 간 API 컨트랙트 정의 완료.
*   **자동화 환경:** `Makefile`을 이용한 API 클라이언트 자동 생성 환경 구축 완료.
*   **현재 단계:** 설계 단계를 넘어 본격적인 **기능 구현(Implementation) 단계** 진입.

## 2. 프론트엔드 작업 방향 논의 (Open Design)
기획서의 핵심인 '음식점 투표, 공유, 비교'를 살리기 위한 3가지 UI/UX 무드보드를 제안합니다. 각 안은 현재 API 스펙(`openapi.yaml`) 기준의 구현 난이도를 함께 고려했습니다.

### 💡 3가지 UI/UX 가시안 (레퍼런스)
*   **A안 (Visual-Heavy & Tasty):** 푸디(Foodie) 인스타형 (예: 망고플레이트, 인스타 스토리 투표)
*   **B안 (Quick & Gamified):** 메뉴 월드컵 / 틴더 스와이프형 (예: 이상형 월드컵, 틴더)
*   **C안 (Map & List):** 철저한 데이터 기반 지도/리스트형 (예: 네이버 지도, 배민 검색결과)

### 📊 디자인 제안별 Trade-off Matrix (BE 논의용)
| 구분 | A안 (인스타형) 📸 | B안 (게임형) 🎮 | C안 (리스트형) 🗺️ |
| :--- | :--- | :--- | :--- |
| **핵심 UX** | 식욕 자극, 시각적 재미 | 빠른 결정, 스와이프 액션 | 꼼꼼한 비교, 정보 탐색 |
| **API 수정 필요도** | 🔴 **높음** | 🟢 **없음 (현재 스펙 유지)** | 🔴 **높음** |
| **BE 추가 요구사항** | - 이미지 URL 필드 추가<br>- 이미지 업로드 또는 외부 크롤링 | - 없음 (토너먼트 시 짝수 검증 정도) | - 위도/경도, 평점, 카테고리 추가<br>- 외부 지도 API 연동 |
| **FE 작업 난이도** | 🟡 중간 (이미지 레이아웃) | 🔴 높음 (제스처, 애니메이션 로직) | 🟡 중간 (리스트 렌더링) |

## 3. 백엔드 논의 사항 (Technical Issues)
*   **인증 및 토큰 유지:** `/enter` 시 발급되는 토큰 유지 방식 (LocalStorage vs Cookie)
*   **실시간성 확보:** 투표 결과(`/results`) 업데이트 방식 (Polling vs SSE/WebSocket)
*   **에러 핸들링:** 비즈니스 에러 발생 시 HTTP Status Code 및 메시지 포맷 통일
*   **데이터 유효성:** 음식점 선택지 텍스트 길이 및 최대 개수 제한
