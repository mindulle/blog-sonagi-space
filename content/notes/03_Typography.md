# 🔠 Vol 3. 타이포그래피 (Typography)

UI 디자인에서 정보 전달의 90%는 텍스트가 담당합니다. 폰트의 크기, 간격, 굵기만으로도 디자인의 완성도(Quality)가 극명하게 갈리며, 텍스트가 완벽하게 다듬어졌을 때 비로소 전체 UI가 견고해집니다.

---

## 1. 텍스트 간격과 밀도 (Spacing & Density)

### 1.1 "자간이 너무 날리네요 / 헐거워요"
*   **시스템 용어**: Tracking / Letter-spacing
*   **현상**: 글자와 글자 사이의 간격(자간)이 너무 넓어 단어의 응집력이 떨어지고 시선이 흩어지는 상태.
*   **실무 솔루션**: 영문과 달리 한글 폰트는 기본 자간이 넓게 설계된 경우가 많아, 디폴트에서 `-2% ~ -5%` (CSS: `letter-spacing: -0.02em;`) 정도 자간을 좁혀서 단어를 단단한 덩어리(Chunk)로 묶어줌.

### 1.2 "줄 간격 때문에 숨 막혀요"
*   **시스템 용어**: Leading / Line-height
*   **현상**: 텍스트 여러 줄이 겹쳐 있을 때, 위아래 줄의 간격(행간)이 너무 좁아 읽기 답답하고 눈이 피로한 현상.
*   **실무 솔루션**: 본문(Body) 텍스트의 행간은 폰트 크기의 **140~160%** (`line-height: 1.5;`)로 설정하고, 제목(Heading)은 한 줄로 읽히게 묶어야 하므로 **110~120%**로 상대적으로 좁게 설정함.

### 1.3 "텍스트 사이에 강이 흐르네요 (Rivers)"
*   **시스템 용어**: Rivers in Justified text
*   **현상**: 텍스트를 양쪽 정렬(Justify) 했을 때, 단어 사이의 공백이 세로로 우연히 이어져 마치 본문 사이에 흰 강물이 흐르는 것처럼 시선이 찢어지는 치명적 현상.
*   **실무 솔루션**: 웹 디자인에서는 특별한 의도가 없다면 무조건 **좌측 정렬(Left-align)**을 원칙으로 하여, 단어 간 공백이 강제 변형되는 것을 막음.

## 2. 시각적 위계와 굵기 (Hierarchy & Weight)

### 2.1 "덩어리감 / 위계(Hierarchy)가 안 잡혀요"
*   **시스템 용어**: Typographic Hierarchy
*   **현상**: 제목, 소제목, 본문의 폰트 크기나 굵기 차이가 적어, 화면을 훑어볼 때 무엇이 제일 중요한 정보인지 한눈에 파악이 안 되는 상태.
*   **실무 솔루션**: 단순 크기 차이뿐 아니라 명도(Contrast)와 굵기(Weight)를 함께 비틂. 
    *   *제목*: 크고 두껍고 짙은 검은색(Bold, `#111`).
    *   *본문*: 적당한 크기, 얇고 약간 회색빛(Regular, `#555`).

### 2.2 "폰트가 찌그러졌어요"
*   **시스템 용어**: Distorted Font / Horizontal Scale (장평) 변형
*   **현상**: 주어진 공간에 긴 글자를 구겨 넣기 위해 텍스트 박스를 강제로 좌우로 쥐어짜서 글자가 세로로 길쭉하게 왜곡된 절대 금기 사항.
*   **실무 솔루션**: 폰트 디자이너가 설계한 본래의 장평(100%)을 절대 건드리지 않음. 글자가 길면 공간을 늘리거나, 줄바꿈을 하거나, 말줄임표(`...`)로 자름.

### 2.3 "시각적으로 헤딩(Heading)이 널뛰네요"
*   **시스템 용어**: Heading Jumps / Inconsistent Hierarchy
*   **현상**: HTML 구조나 마크업에서 `H1` 다음 `H2`가 나오지 않고 갑자기 `H4`로 건너뛰어 시각적 스케일 시스템(Type Scale)이 무너진 상태.
*   **실무 솔루션**: `12-14-16-20-24-32-48px` 등 일정한 기하급수적 수학 비율(Major Third, Perfect Fourth 등)을 따르는 টাই프 스케일 토큰을 철저히 준수.

## 3. 폰트 렌더링 이슈 (Rendering & Bugs)

### 3.1 "폰트에 안티(Anti)가 먹어서 흐려요"
*   **시스템 용어**: Font Smoothing / Anti-aliasing Rendering
*   **현상**: 브라우저나 OS(Mac vs Windows)의 텍스트 렌더링 엔진 차이로 인해, 디자인 시안(Figma)에서는 얇고 선명했던 폰트가 웹에 올리면 두껍고 흐릿하게 렌더링되는 현상.
*   **실무 솔루션**: CSS에 `-webkit-font-smoothing: antialiased;` 속성을 추가하여 Mac과 아이폰 환경에서 폰트가 지나치게 뚱뚱해지는 것을 보정함.

### 3.2 "시스템 폰트(명조/고딕)가 안 맞아요"
*   **시스템 용어**: Fallback Fonts / Sans-serif vs Serif
*   **현상**: 사용자 PC에 웹폰트가 다운로드되지 않아, 브라우저가 강제로 기본 폰트(바탕체나 굴림체)로 렌더링하여 디자인 톤앤매너가 완벽하게 박살 난 상태.
*   **실무 솔루션**: CSS `font-family`에 웹폰트를 맨 앞에 선언하고, 그 뒤로 `Apple SD Gothic Neo, Noto Sans KR, sans-serif` 등 안전한 Fallback 폰트를 촘촘히 엮어 시스템 방어망 구축.

## 4. [Data Sourced] 폰트 디자이너 특화 타이포그래피 은어
*   **"커닝(Kerning) 밸런스가 무너졌어요"**
    *   *의미*: 전체 글자 간격(자간/Tracking)이 아니라, 특정 두 글자(예: 'T'와 'y' 또는 'W'와 'A') 사이의 물리적 공백이 유독 넓어 보여서 형태가 찢어지는 현상.
    *   *액션*: 두 글자 사이의 간격만 콕 집어서 시각적으로 좁혀주는 커닝 보정.
*   **"시각 삭제 (Optical Ink Trap / Overshoot)"**
    *   *의미*: 둥근 알파벳(o, c, a)이나 한글의 'ㅇ'이 직선 글자와 나란히 있을 때 작아 보이는 것을 막기 위해 위아래로 살짝 튀어나가게(Overshoot) 폰트가 설계된 원리.
    *   *액션*: 디자이너는 폰트를 화면에 배치할 때 이 '시각 삭제' 영역을 고려하여 그리드에 정렬해야 함.
*   **"글줄이 길면 행간을 열어주세요"**
    *   *의미*: 텍스트 단락(문단)의 가로 너비가 길어질수록, 읽고 나서 다음 줄로 눈이 찾아가기 힘들어지므로 행간(Line-height)을 더 넓혀야(160% 이상) 가독성이 확보된다는 법칙.
