# 🌿 팀 협업 및 브랜칭 전략 (Git Strategy)

우리 프로젝트는 6/7 마감을 위한 **속도와 안정성**을 최우선으로 하며, 다음과 같은 단순화된 브랜칭 전략을 사용합니다.

## 1. 브랜치 구조
- **`main`**: 상시 배포 가능한 상태를 유지하는 브랜치. Azure App Service와 연결되어 푸시 시 즉시 배포됩니다.
- **`feature/[기능명]`**: 각 이슈별 독립적인 개발 브랜치. (예: `feature/kakao-login`, `feature/swipe-ui`)

## 2. 작업 프로세스 (Work-flow)
1. **이슈 할당**: 깃허브 이슈에서 자신의 작업을 확인합니다.
2. **브랜치 생성**: `main` 브랜치로부터 최신 코드를 받아 자신의 기능 브랜치를 만듭니다.
3. **개발 및 커밋**: 작업 단위별로 명확한 커밋 메시지를 작성합니다.
4. **Pull Request (PR)**: 작업 완료 후 `main` 브랜치로 PR을 날립니다.
5. **리뷰 및 병합**: 팀원(또는 혜성님)의 최소 1인 승인 후 `main`에 병합합니다.

## 3. 커밋 메시지 규칙 (Commit Convention)
- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정 (위키, 가이드 등)
- `refactor`: 코드 리팩토링
- `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)

---
## 💡 주의 사항
- **직접 푸시 금지**: `main` 브랜치로의 직접적인 `git push`는 지양하고, 반드시 PR을 거쳐야 합니다. (긴급 수정 제외)
- **동기화**: 작업 시작 전 항상 `git pull origin main`을 통해 최신 상태를 유지하세요.
