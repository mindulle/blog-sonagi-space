# [Assignment 3] Wittgenstein Revisited

### 1. 언어의 문법이 규정하는 사고의 경로

루트비히 비트겐슈타인은 "내 언어의 한계는 내 세계의 한계다"라고 말하며, 우리가 사용하는 언어의 틀이 곧 우리가 세상을 파악하고 사고하는 한계를 결정짓는다고 주장했습니다. 프로그래밍의 영역에서도 공학자는 자신이 사용하는 언어의 패러다임 안에서 문제를 정의하고 해결책을 구상하게 됩니다. 예를 들어, 과거 C 언어와 같은 절차적 언어에 익숙한 공학자는 문제를 '메모리 제어와 순차적 단계'로 인식하는 경향이 강합니다. 반면, 현대의 고차원 언어나 선언적 문법을 익힌 공학자는 문제를 '데이터 흐름'이나 '상태의 정의'라는 관점에서 바라보게 되며, 이는 곧 설계할 수 있는 아키텍처의 한계를 확장하는 결과로 이어집니다.

### 2. 비동기 이벤트 중심 사고와 spells-bot 프로젝트

제가 개발한 디스코드 봇인 `spells-bot`은 JavaScript(Node.js)의 비동기적 특성을 활용하여 '이벤트 중심'의 세계관을 실천한 사례입니다. 전통적인 C 언어의 사고방식에서는 특정 작업이 완료될 때까지 프로그램이 멈춰 서는 '동기적(Synchronous)' 흐름이 지배적이지만, Node.js 환경에서는 수많은 사용자의 요청에 동시다발적으로 반응하는 '비동기적(Asynchronous)' 사고가 필수적입니다. 아래의 코드 예시는 텍스트 기반 언어 내에서 비동기 제어 문법(`async/await`)을 통해 어떻게 실시간 상호작용이라는 세계를 구축하는지 보여줍니다.

> **[코드 예시 1: Node.js의 비동기 이벤트 처리]**

```jsx
// spells-bot: Discord.js를 이용한 명령어 수신 로직
client.on('interactionCreate', async interaction => {
  if (!interaction.isChatInput()) return;
  
  // /snippet 명령어를 받으면 외부 n8n 워크플로우로 처리를 위임함
  if (interaction.commandName === 'snippet') {
    const query = interaction.options.getString('query');
    // 비동기 호출을 통해 '기다림'의 정체를 해결하고 세계를 외부로 확장
    const response = await axios.post('https://n8n.sonagi.space/workflow/XpeYlYfcmSDAboCe', { query });
    await interaction.reply(`결과를 찾았습니다: ${response.data.results.length}건`);
  }
});
```

### 3. 시각적 프로그래밍 언어를 통한 설계 지평의 확장

프로젝트 과정에서 n8n이라는 노코드/로우코드 도구를 도입했습니다. `spells-bot`에서 송신한 데이터를 처리하는 n8n 워크플로우는 `IF` 노드와 `Supabase` 노드 등을 통해 로직을 시각적으로 구조화합니다. 이는 복잡한 조건 분기(`Has Language Filter?`)를 코드가 아닌 '흐름(Flow)'으로 인식하게 하며, 공학자가 시스템의 전체적인 맥락을 한눈에 파악할 수 있게 돕습니다. 비트겐슈타인의 주장처럼, 텍스트(JavaScript)에서 시각적 도구(n8n)로 언어적 수단을 확장함으로써, 저는 더 복잡한 자동화 아키텍처를 설계하고 제어할 수 있는 새로운 세계에 진입하게 된 것입니다.

### 4. 결론: 공학자에게 언어가 갖는 의미

결국 새로운 프로그래밍 언어나 도구를 배우는 것은 단순히 문법을 익히는 과정이 아니라, 문제를 해결하는 새로운 '세계관'을 습득하는 과정입니다. 특정 언어가 제공하는 추상화의 수준이 곧 공학자가 설계할 수 있는 아키텍처의 한계가 됩니다. 저는 `spells-bot` 개발 과정에서 Node.js, n8n, SQL이라는 서로 다른 언어적 세계를 연결하며 저의 공학적 지평을 넓힐 수 있었습니다. 풍부한 언어적 도구를 갖추는 것이야말로 공학적 사고의 지평을 넓히는 가장 강력한 방법이라는 비트겐슈타인의 통찰은 현대 소프트웨어 공학에서도 여전히 유효하다고 생각합니다.
