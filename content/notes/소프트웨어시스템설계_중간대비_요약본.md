# ☁️ 소프트웨어시스템설계 중간고사 단권화 노트

> 본 문서는 수업 강의 자료(소프트웨어시스템설계 첫 수업, 리팩토링, 디자인 패턴, 단위 테스트 등)를 기반으로 작성된 중간고사 대비 요약본입니다.

## 📌 1. 소프트웨어시스템설계 개요 및 목표
* **과목 목표**: 소프트웨어 공학의 기본 개념을 바탕으로, 실제 현장에서 소프트웨어를 개발하는 데 필요한 설계, 리팩토링(Refactoring), 디자인 패턴(Design Pattern) 적용 능력을 실무적으로 갖추는 것.
* **소프트웨어 설계의 특성**: 
  * "정답은 없다." (요구사항, 특히 성능, 유지보수성, 가용성 등 비기능적 품질에 따라 최적의 설계가 달라짐)
  * 최근 AI(LLM)의 발전으로 코딩 작업은 많이 자동화되었으나, '설계'는 아직 AI가 완전히 대체하기 어려운 인간 고유의 영역으로 남아 있음. 따라서 설계 능력을 갖추는 것이 개발자로서 생존력을 높이는 길임.

## 📌 2. 리팩토링 (Refactoring)
* **리팩토링의 정의**: 겉으로 드러나는 소프트웨어의 기능적인 동작은 변경하지 않으면서, 내부 구조를 개선하여 이해하기 쉽고 수정하기 쉬운 코드로 만드는 과정.
* **단위 테스트 (Unit Test)의 중요성**: 리팩토링 과정에서 기존 기능이 훼손되지 않음을 보장하기 위해 자동화된 단위 테스트(예: JUnit)가 필수적임.
* **코드 스멜 (Code Smell)**: 리팩토링이 필요한 코드의 징후. 각 스멜에 맞는 적절한 리팩토링 기법을 매칭하여 적용해야 함.
* **Refactoring to Design Patterns**: 코드를 일차적으로 깔끔하게 정리한 뒤, 더 나아가 검증된 구조인 디자인 패턴이 드러날 수 있도록 코드를 개선하는 심화 과정.

## 📌 3. 디자인 패턴 (Design Pattern) Overview
* **디자인 패턴의 의의**: 반복적으로 발생하는 설계 문제에 대한 검증된 해결책.
* **주요 패턴 분류**:
  1. **생성 패턴 (Creational Pattern)**: 인스턴스 생성 과정을 추상화. (예: Factory Method, Abstract Factory, Singleton, Builder)
  2. **구조 패턴 (Structural Pattern)**: 클래스나 객체들을 조합하여 더 큰 구조를 만듦. (예: Adapter, Bridge, Composite, Decorator, Facade)
  3. **행위 패턴 (Behavioral Pattern)**: 객체나 클래스 사이의 알고리즘이나 책임 분배에 관련된 패턴. (예: Strategy, Command, Observer, State, Template Method)
