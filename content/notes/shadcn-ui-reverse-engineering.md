# shadcn/ui 역기획 분석

## 1. 서비스 및 시스템 개요
- **대상**: [shadcn/ui](https://ui.shadcn.com/)
- **분석 목적**: 1인 및 소규모 팀이 "소유"할 수 있는 디자인 시스템 구조 분석 및 LLM-Wiki 적용 방안 모색.
- **핵심 철학**: "라이브러리가 아니라, 컴포넌트의 모음이다." (Not a component library, but a collection of re-usable components).

## 2. 시스템 구조 분석 (Technical Architecture)
- **기반 기술**: 
    - **Radix UI**: 접근성(Accessibility)과 핵심 로직(Headless) 담당.
    - **Tailwind CSS**: 스타일링 및 디자인 토큰(CSS Variables) 담당.
    - **Lucide React**: 아이콘 세트.
- **배포 방식**: npm 설치가 아닌 CLI(`npx shadcn-ui@latest add [component]`)를 통한 소스코드 직접 주입.

## 3. 리서치 및 디자인 가이드 준수 체크 (Audit)
*[[Design-System-Research-Guide|디자인 시스템 리서치 가이드]] 기반 분석*

### 🎨 디자이너/개발자 워크플로우
- **재사용 vs 수정**: 모든 코드가 프로젝트 내부에 위치하므로, 라이브러리 업데이트에 따른 Breaking Change 걱정 없이 자유롭게 커스텀 가능.
- **핸드오프**: Figma 변수(Variables)와 코드의 CSS 변수가 1:1로 매칭되도록 설계됨.

### 💻 시스템 디자인 (확장성)
- **성능**: 런타임 자바스크립트가 거의 없으며, 필요한 컴포넌트의 코드만 프로젝트에 포함되므로 번들 사이즈가 최적화됨.
- **일관성**: `globals.css`의 CSS 변수를 통해 전역 테마(Primary, Secondary, Radius 등)를 관리.

## 4. 핵심 컴포넌트 분석 (Live Audit)
*Storybook 및 공식 문서를 통한 분석*
- **Button**: 다양한 Variant(ghost, outline, link 등)를 `class-variance-authority (cva)`로 관리하는 방식.
- **Form**: `react-hook-form` 및 `zod`와의 긴밀한 통합을 통해 유효성 검사 로직을 시스템화함.

## 5. 인사이트 및 LLM-Wiki 적용점
- **Atomic Note 연계**: shadcn/ui가 컴포넌트를 독립된 파일로 관리하듯, 우리 위키도 개념을 원자적 노트(`_concepts/`)로 관리하여 필요할 때 조합하는 방식 유지.
- **CLI 기반 관리**: 위키의 `00_System/scripts/`에 shadcn/ui와 같은 자동화 스크립트를 보강하여 문서 템플릿 주입 프로세스 고도화.
- **디자인 토큰**: `00_System/design-tokens/variables.css`를 shadcn/ui의 테마 시스템 구조로 재편성.
